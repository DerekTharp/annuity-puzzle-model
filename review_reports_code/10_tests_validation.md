# Test Suite and Numerical Validation Review

## 1. Summary Assessment
The codebase has a respectable amount of coverage for a structural economics project: utility functions, health dynamics, limiting cases, simulation, decomposition, and welfare all have some tests. The problem is that the validation layer is still too loose for the precision of the paper’s headline claim. In particular, the phase-4 regression path does not actually exercise the health-aware population ingestion in the decomposition test, and the Euler/convergence diagnostics show enough instability that I would not fully trust the production `80x30x3x46` solution without further tightening.

Overall, this is a solid but not yet submission-grade testing package. I would treat it as functional and promising, but still needing a sharper regression suite and stronger numerical validation before I would rely on the reported 5.3% baseline ownership as a locked result.

## 2. Specific Findings
1. **[high] Phase 4 regression tests do not actually validate the health-aware decomposition path.** The main decomposition test in [test/test_phase4.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/test/test_phase4.jl#L219) constructs `pop` with only wealth, income, and age, then passes that 3-column matrix into `run_decomposition` at [test/test_phase4.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/test/test_phase4.jl#L241). That means the health-aware code path can silently fall back to Fair health, because the evaluator in [src/decomposition.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/src/decomposition.jl#L233) and [src/wtp.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/src/wtp.jl#L528) defaults missing health to `2`. The small `solve_and_evaluate` smoke test only uses a synthetic all-Fair health column, so a bug in observed-health ingestion would still pass these tests.

2. **[high] The Euler and convergence diagnostics do not yet justify trusting the production numerical solution at the stated precision.** The reported residuals in [tables/csv/euler_residuals.csv](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/tables/csv/euler_residuals.csv#L1) include a maximum near `0.99` and about `6%` above `5%`, while [tables/csv/convergence_diagnostics.csv](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/tables/csv/convergence_diagnostics.csv#L1) shows ownership moving from `21.38%` at 9-node quadrature to `16.78%` at the 120x50, 11-node reference. The diagnostics script itself, [scripts/run_euler_diagnostics.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/scripts/run_euler_diagnostics.jl#L67), still reports those same oscillations across node counts. Even though the appendix describes the 9-node rule as well-converged at [paper/appendix.tex](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/paper/appendix.tex#L168) and [paper/appendix.tex](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/paper/appendix.tex#L410), the evidence is not yet strong enough for a high-precision quantitative claim.

3. **[medium] The suite does not directly regression-test the published 5.3% baseline ownership result.** The closest Phase 4 test only checks that ownership is at least `0.3` and declines monotonically in the decomposition at [test/test_phase4.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/test/test_phase4.jl#L252), while the Lockwood and welfare tests also rely on broad ranges rather than the exact headline value at [test/test_lockwood.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/test/test_lockwood.jl#L40) and [test/test_welfare.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/test/test_welfare.jl#L70). That leaves the key paper result vulnerable to silent drift.

4. **[medium] The convergence diagnostics are not run on the exact production SS specification.** The appendix notes that the convergence tables use mean Social Security income rather than the quartile-specific SS assignment in the production decomposition at [paper/appendix.tex](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/paper/appendix.tex#L168), and the diagnostic scripts implement that simplification explicitly at [scripts/run_euler_diagnostics.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/scripts/run_euler_diagnostics.jl#L27) and [scripts/grid_convergence_full.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/scripts/grid_convergence_full.jl#L83). That is fine as a robustness check, but it does not fully certify the exact object used for the headline 5.3% result.

5. **[medium] The moment-validation script is mostly a calibrated sanity check rather than a genuinely external holdout test.** The script hard-codes empirical targets from the same literature used elsewhere in the paper at [scripts/run_moment_validation.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/scripts/run_moment_validation.jl#L86) and evaluates a representative initial state rather than a held-out sample at [scripts/run_moment_validation.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/scripts/run_moment_validation.jl#L72). The appendix also presents the exercise as an external check at [paper/appendix.tex](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/paper/appendix.tex#L349), but in practice it is closer to a structured calibration sanity check than a strict out-of-sample validation.

6. **[minor] The limiting-case tests are too weak to fully protect the model’s edge cases.** The “certain lifespan” test in [test/test_limiting_cases.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/test/test_limiting_cases.jl#L84) only checks for `NaN`s, and the zero-wealth test in [test/test_limiting_cases.jl](/Users/derektharp/Documents/_Projects/26.29%20Annuity%20Puzzle%20Simulation/test/test_limiting_cases.jl#L105) mostly checks the grid rather than asserting the expected annuitization outcome. These are useful smoke tests, but they do not yet pin the intended economics tightly enough.

## 3. Concrete Fix Recommendations
1. Add a regression test that feeds a real 4-column population with observed health into the phase-4 decomposition path, and assert that changing health data changes the result. That would close the current false-positive hole in the health-aware pipeline.

2. Add an exact baseline regression assertion for the published 5.3% ownership, with a small tolerance, and run it against the final production configuration. That would make the key paper output much harder to drift accidentally.

3. Tighten the numerical validation by either improving quadrature resolution or lowering the precision of the claims until the residuals and convergence tables stabilize. If 11-node quadrature remains a materially different answer, the appendix should say so plainly.

4. Keep the current moment-validation script, but relabel it as a calibration sanity check unless you add a true holdout or external benchmark. If you want it to function as validation, add at least one genuinely independent target.

5. Strengthen the limiting-case tests so they assert the expected policy outcomes, not just the absence of `NaN`s. The edge-case tests should prove the economics, not only the plumbing.
